# 🏭 MES (Manufacturing Execution System)

제조 현장의 핵심 업무를 하나의 웹 시스템에서 처리하는 MES 프로젝트입니다.
입출고·재고 관리부터 생산계획, 작업지시, 품질검사, 설비관리, 기준정보까지 제조 실행에 필요한 전 과정을 REST API 기반으로 구현했습니다.

## 📌 프로젝트 정보

- **기간** : 2026.04.02 ~ 2026.04.21
- **인원** : 5명 (팀 프로젝트)
- **담당 역할** : 입출고 등록 / 재고 관리 / 공지사항 / 건의사항 풀스택 구현

## 🛠 기술 스택

- **Frontend** : HTML5, CSS3, JavaScript, AJAX, JSP
- **Backend** : Java, Servlet, REST API
- **Database** : Oracle
- **Server** : Apache Tomcat

## ✨ 전체 기능

| 모듈 | 설명 |
|---|---|
| 인증 / 권한 | 로그인, 회원가입, 마이페이지, 비밀번호 변경, 권한 관리 |
| 대시보드 | 주요 현황 지표 요약 |
| 공지사항 | 공지 게시판 (작성·조회·수정·삭제, 검색, 파일 첨부) |
| 건의사항 | 건의 게시판 (댓글·대댓글, 처리 상태 관리, 권한별 열람 제한) |
| 입출고 / 재고 | 입고·출고 등록, LOT 단위 관리, 재고 현황 조회, 안전재고 설정 |
| 생산계획 / 작업지시 | 생산 계획 수립, 작업지시(WO) 등록·관리, LOT 추적 |
| 품질관리 | 품질 검사 등록, 불량 유형 관리, 불량품 처리 |
| 설비관리 | 설비 현황 조회, 설비 가동 로그 기록·수정 |
| 리포트 | 생산 실적·불량 리포트 |
| 기준정보 | BOM, 품목, 공정, 거래처 마스터 데이터 관리 |

데이터베이스는 품목·LOT·입출고·작업지시·품질·설비 등 총 27개 테이블로 설계했습니다.

## 🙋 담당 기능 상세

### 입출고 등록

- 입고와 출고를 하나의 화면에서 등록하고, 입출고 사유·거래처·담당자·일자를 함께 기록합니다.
- 입고 시 LOT 번호와 유효기간을 부여하여 LOT 단위로 수량을 추적합니다.
- 출고 등록 시 AJAX로 출고 가능한 LOT 목록을 비동기 조회하여 페이지 새로고침 없이 선택할 수 있게 했습니다.
- 출고 수량이 현재 재고를 초과하면 등록을 차단하고 현재 재고 수량을 안내하는 검증 로직을 넣어 재고가 음수가 되는 상황을 방지했습니다.
- 입출고 일시는 KST 기준으로 변환하여 저장해 시간대 불일치 문제를 차단했습니다.

### 재고 관리

- 품목별 재고 현황을 페이징 목록으로 제공하고, 품목 그룹·키워드·재고 상태(정상/부족) 조건으로 필터링할 수 있습니다.
- 화면 상단에 전체·정상·부족 재고 건수를 요약해 한눈에 재고 상태를 파악할 수 있게 했습니다.
- 품목별 안전재고 수량을 AJAX로 즉시 수정할 수 있으며, 안전재고 미달 품목은 부족 재고로 분류됩니다.

### 공지사항

- 공지 게시글의 작성·조회·수정·삭제와 키워드 검색, 페이지 그룹 단위 페이징을 구현했습니다.
- 쿠키를 활용해 같은 사용자가 반복 조회해도 조회수가 중복 증가하지 않도록 처리했습니다.
- 파일 첨부 기능을 구현했으며(업로드 용량 제한 10MB), 다운로드 시 원본 파일명이 유지되도록 인코딩을 처리했습니다.

### 건의사항

- 건의 글 등록·조회·삭제와 댓글, 대댓글 기능을 구현했습니다.
- 상세 페이지는 작성자 본인 또는 관리자 권한 이상만 열람할 수 있도록 세션 기반 접근 제어를 적용했습니다.
- 작성자가 아닌 사람이 댓글을 달면 처리 상태가 자동으로 "답변 완료"로 변경되고, 관리자가 직접 처리 완료로 전환할 수도 있습니다.
- 작성 후 10분이 지나면 삭제할 수 없도록 제한을 두어 답변 중인 글이 사라지는 문제를 방지했습니다.

## 🔥 트러블슈팅

### ERD 변경으로 인한 연쇄 수정

- **문제** : 개발 도중 필요한 컬럼이 누락되어 있거나 FK 관계가 잘못 연결된 것이 발견되어 ERD를 반복적으로 수정하게 되었고, 그때마다 관련 DAO와 SQL까지 연쇄적으로 수정해야 했습니다.
- **해결** : 테이블 변경 시 영향을 받는 코드 목록을 먼저 정리한 뒤 일괄 수정하고, 이후의 스키마 변경은 팀에 공유한 후 반영하는 방식으로 바꿨습니다.
- **배운 점** : 설계 단계에서 ERD를 충분히 검증하는 비용이 구현 단계에서 수정하는 비용보다 훨씬 작다는 것을 체감했습니다.

### 시연 직전 통합 테스트

- **문제** : 통합 테스트 일정을 시연 하루 전으로 잡는 바람에 발견된 버그를 밤새 수정해야 했습니다.
- **배운 점** : 테스트는 마지막에 몰아서 하는 것이 아니라 기능 단위로 마일스톤마다 분산해야 하며, 일정에는 반드시 버퍼가 필요하다는 것을 배웠습니다.

## 💡 배운 점

- 게시판·입출고·재고 기능을 직접 설계하고 구현하면서 CRUD의 개념을 실제 데이터 흐름 속에서 정확히 이해하게 되었습니다.
- 화면(JSP/AJAX)부터 서블릿, SQL까지 풀스택으로 구현하며 요청이 처리되는 전체 흐름을 익혔습니다.
- 재고 검증, 권한 제어, 조회수 중복 방지처럼 "동작하는 기능"을 넘어 "안전하게 동작하는 기능"을 만드는 관점을 갖게 되었습니다.
